Api 与 Spi

  • Api ( Application Programming Interface ) :大多数情况下，都是实现方来制定接口并完成对接口的不同实现，调用方仅仅依赖却无权选择不同实现。
  • Spi ( Service Provider Interface ) :而如果是调用方来制定接口，实现方来针对接口来实现不同的实现。调用方来选择自己需要的实现方。
  • 接口位置
    • 位于调用方的包 (SPI)
    • 插件模式 (数据库驱动Driver,日志Log,dubbo扩展节点开发) 一种服务发现机制,为了实现在模块装配的时候可以不在程序里动态指明.(模块化设计,将配置的控制权移到程序之外) java.util.ServiceLoader
    • 位于实现方的包 (API)
    • 位于独立的包
      • 在一个上下文是API,在另一个上下文是SPI

JVM关闭钩子

Runtime.getRuntime().addShutdownHook();

  • 在jvm中添加一个关闭钩子,当jvm关闭的时候回执行系统中所有添加的钩子,当系统执行完这些钩子后,jvm才会关闭,这些钩子在jvm关闭时进行垃圾清理,对象销毁.
  • jvm关闭的三种方式
    • 正常关闭:当最后一个守护线程结束或者调用了System.exit(),或通过其他特定平台的方法关闭(发送SIGINT,SIGTERM信号等)
    • 强制关闭:调用Runtime.getRuntime().halt();或者在操作系统中直接kill(发送SIGKILL信号)掉jvm进程
    • 异常关闭:运行中遇到RuntimeException等
  • 注意:
    • hook线程会延迟JVM的关闭时间,所以尽可能减少执行时间
    • 关闭钩子中不要调用System.exit(),会卡住JVM的关闭过程,但可以调用Runtime.getRuntime().halt();
    • 不能在钩子中进行钩子的添加和删除,会抛IllegalStateException
    • 在system.exit()后添加的钩子无效,因为此时jvm已经关闭了
    • 当JVM收到SIGTERM命令(比如操作系统在关闭时)后,如果钩子线程在一定时间没有完成,那么Hook线程可能在执行过程中被终止
    • Hook线程也会抛错,若未捕获,则钩子的执行序列会被停止

ClassLoader

  • Bootstrap => Ext => App => 自定义

    • 双亲委托: 由下到上寻找(起点从父加载器开始,终点是其本身),顶层加载不了再交给下层加载,底层找不到,报ClassNotFound. 委托自下向上,查找过程自上向下.
    • BootstrapClassLoader是用c++编写的,无法在java代码中获取它的引用,这个加载器会返回null.
    • BootstrapClassLoader加载路径sun.boot.class.path默认是/jre/lib/rt.jar, resources.jar,charset.jar和class文件等; 可以通过java -Xbootclasspath/a:path追加文件到其默认路径中
    • ExtClassLoader加载路径java.ext.dirs默认是/jre/lib/ext/*.jar和class文件,可以-D java.ext.dirs追加指定的目录
    • AppClassLoader 加载路径 java.class.path,想要loadClass成功需要完整的包名,
    • ClassLoader加载源码:loadClass().

    • ClassLoader的parent赋值:

      • 由外部类创建ClassLoader时直接指定一个ClassLoader为parent
      • 由getSystemClassLoader()方法生成，也就是在sun.misc.Laucher通过getClassLoader()获取.即AppClassLoader(一个ClassLoader创建时如果没有指定parent，那么它的parent默认就是AppClassLoader)

  • 自定义ClassLoader,需重写

    .

    1. 继承ClassLoader
    2. 覆写findClass()方法
    3. 在findClass()方法中调用defineClass().
  • 自定义ClassLoader常见用途:
    • 跨路径加载 (ps,后缀也不需要是class,最终是读二进制流)
    • class文件加密,在ClassLoader中解密加载
  • Context ClassLoader 线程上下文类加载器
    • 子线程的ContextClassLoader是AppClassLoader
    • AppClassLoader加载不了父线程当中已经加载的class内容
      • 可以在子线程开头设置Thread.currentThread().setContextClassLoader();
    • 运用:
      • 适应Web服务框架如Tomcat.同一份class文件在不同的加载器下生成的类是不相等的
      • 线程相关,可以获取和设置,可以绕过双亲委托的机制
