Java基础

  • 当对象中自定义了 writeObject 和 readObject 方法时，JVM 会调用这两个自定义方法来实现序列化与反序列化。( ArrayList 只序列化了被使用的数据。)
  • hashCode()值影响HashSet等,添加时会将不同值识别为不一样的实例,即使equals()相等的情况.
  • 理想的散列函数应当具有均匀性，即不相等的实例应当均匀分布到所有可能的散列值上。这就要求了散列函数要把所有域的值都考虑进来，可以将每个域都当成 R 进制的某一位，然后组成一个 R 进制的整数。R 一般取 31，因为它是一个奇素数，如果是偶数的话，当出现乘法溢出，信息就会丢失，因为与 2 相乘相当于向左移一位。
  • toString() 返回 @ 后面的数值为散列码的无符号十六进制表示

深入理解Java虚拟机

发表于 2018-06-25 | 分类于 读书笔记 | | 阅读次数:

内存区域与内存溢出异常

程序计数器

  • 当前线程执行的字节码的行号指示器
  • 线程私有
    • (Java 多线程是通过线程轮流切换并分配处理器执行时间来实现的)
  • ==唯一一个没有OOM的区域==

Java虚拟机栈

  • Java方法执行的内存模型
  • 线程私有
  • 栈帧
    • 用于存储局部变量表,操作数栈,动态链接,方法出口等信息
    • 局部变量表 (所需内存在编译期间完成分配)
      • 基本数据类型
      • 对象引用
      • returnAddress类型
  • StackOverflowError
    • 线程请求的栈深度大于虚拟机所允许的深度
  • OutOfMemoryError
    • 虚拟机栈扩展时无法申请到足够的内存

本地方法栈

  • 也有SOF和OOM
    • 内存泄漏(Memory leak)
      • 指程序申请内存后,无法释放已申请的内存空间
    • 内存溢出(OOM)
      • 指程序申请内存时,没有足够的内存供申请者使用
    • (内存泄漏的堆积会造成内存溢出)

Java堆

  • 线程共享
  • 存放对象实例
  • 划分
    • 内存回收角度
      • 新生代
      • 老年代
    • 内存分配角度
      • 多个线程私有的分配缓冲区(Thread Local Allocation Buffer,TLAB)
  • 内存空间逻辑上连续即可
  • 有OOM

方法区

  • 线程共享
  • 存储已被虚拟机加载的类信息,常量,静态变量,即时编译器编译后的代码等数据
  • 堆的一个逻辑部分,但别名 Non-Heap
  • 方法区 不等于 永久代 (只是HotSpot把GC分代收集扩展至方法区,1.7开始移除)
  • 虚拟机规范对方法区限制非常宽松,可以选择不实现垃圾收集
  • 有OOM

运行时常量池

  • 方法区的一部分
    • ==常量池==(是Class文件中的一部分,Java虚拟机对Class文件的每一部分有严格要求,每个字节存储哪种数据)用于存放编译期生成的各种字面量和符号引用,这部分内容将在类加载后进入方法区的==运行时常量池==(Java虚拟机规范没有做任何细节要求)
    • Java并不要求常量一定只有编译期才能产生,运行期间也可能有新的常量放入运行时常量池,如 String.intern()
    • 有OOM

直接内存

  • 堆外内存,通过堆中对其的引用操作(避免来回复制数据)
  • NIO,基于 Channel 与 Buffer 的 I/O 方式, DirectByteBuffer
  • 有OOM (直接内存分配不受Java堆大小的限制,配置参数时要考虑预留)

对象的创建

  1. 类加载检查(常量池中定位符号引用)
  2. 分配内存
    • Java堆是否规整:由其采用的垃圾收集器是否带有压缩整理功能(compact)决定
    • 堆内存规整:指针碰撞(已使用与空闲的内存中间放着指针作为分界点,操作指针)
    • 堆内存不规整:空闲列表(维护一张记录可用内存的表)
    • 并发情况下并非线程安全
      1. 同步锁定
        • CAS配上失败重试的方式保证更新操作原子性
      2. TLAB(本地线程分配缓冲 Thread Local Allocation Buffer) + 同步锁定
        • 把分配动作按照线程划分在不同的空间中进行
  3. 初始化零值(不含对象头)
  4. 必要设置(对象头信息)

对象的内存布局

  • 对象头(Header)
    • 对象自身的运行时数据:哈希码,GC分代年龄,锁状态标志,线程持有的锁,偏向线程ID,偏向时间戳等 (官方称这部分为 Mark World)
    • 类型指针:对象指向它的类元数据的指针 (查找对象的元数据信息并不一定要经过对象本身)
  • 实例数据:程序代码中所定义的各种类型的字段内容
  • 对齐填充:这部分不是必然存在的,仅仅起着占位符的作用(HotSpot VM 要求对象的起始地址必须是8字节的整数倍)

对象的访问定位

  • Java程序通过==栈==上的reference数据来操作堆上的具体对象
  1. 句柄: reference不需要修改
    • ==堆==中划出一部分内容作为句柄池,reference存储句柄地址,而句柄中包含了对象实例数据与类型数据各自的具体地址信息
    • 好处:在对象被移动(GC时),只会改变句柄中的实例数据指针,而reference本身不需要修改
  2. 直接指针: 少一次指针定位,速度更快
    • HotSpot采用此种方式

垃圾收集器与内存分配策略

回收相关

  • 引用计数算法 (很难解决对象间相互引用的问题,未被JVM采用)
  • 可达性分析算法 (通过一系列 GC Roots 作为起始点)

引用

  • 强引用:永远不会被回收掉
  • 软引用:OOM之前进行二次回收
  • 弱引用:下一次GC前回收,不管内存是否足够
  • 虚引用:GC时会有一个系统通知(虚引用无法获得一个对象实例)

对象死亡

  • 对象的真正死亡至少要经历==两次==标记过程
    1. 与GC Roots 没有相链接的引用链,会被第一次标记并进行一次筛选(没有覆盖finalize()或finalize()已被虚拟机调用过),将其放入F-Queue中
    2. GC对F-Queue进行第二次标记. (finalize()是对象在被GC时自我拯救的最后唯一机会,一个对象的finalize()最多只会被系统自动调用一次)

回收方法区

  • 永久代的垃圾收集主要回收两部分内容:废弃常量和无用的类
  • 判断一个类是否是无用的类,需同时满足3条件才==可以==被回收,并非一定回收,取决于jvm参数设置.
    • 该类所有的实例都已经被回收,即Java堆中不存在该类的任何实例
    • 加载该类的ClassLoader已经被回收
    • 该类对应的java.lang.Class对象没有在任何地方被引用,无法在任何地方通过反射访问该类的方法
  • 在大量使用反射,动态代理,CGLib等ByteCode框架,动态生成JSP及OSGI这类频繁自定义ClassLoader的场景都需要虚拟机具备类卸载的功能,以保证永久代不会溢出

垃圾收集算法

  • 标记-清除(Mark-Sweep)
    • 两点不足:效率问题;空间问题(清除后产生大量不连续的内存碎片,分配大对象时,无法找到足够的连续内存会提前触发另一次垃圾收集动作)
  • 复制算法
    • 商业虚拟机多采用此种算法来回收==新生代==
    • 新生代 Eden:Survivor:Survivor = 8:1:1; 每次使用Eden和其中一块Survivor,GC时把这两块区域中存活的对象复制到剩下的Survivor. 当Survivor空间不够用时,需要依赖==老年代==进行分配担保(Handle Promotion)
    • 不足:在对象存活率较高时要进行较多的复制操作,效率会变低;不浪费空间的情况下,需要额外的空间进行分配担保
  • 标记-整理(Mark-Compact)
    • 针对于老年代

多线程

发表于 2017-10-31 | 分类于 碎片 | | 阅读次数:
  • 多线程的优点
    1. 资源利用率更好
      • eg:对多个文件的读取处理
    2. 程序设计在某些情况下更简单
      • eg:不再需要记录文件读取和处理的状态
    3. 程序响应速度更快
      • eg:服务器应用,监听线程负责监听请求,工作线程负责处理响应请求
  • 多线程的代价
    1. 设计更复杂
      • eg:线程交互,同步等
    2. 上下文切换的开销
      • eg:切换线程时,需要存储当前线程的本地数据,程序指针等
    3. 增加资源消耗
      • 除了cpu,线程还需要一些内存来维持它本地的堆栈.
